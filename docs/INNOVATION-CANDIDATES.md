## 0. 판단의 전제 (가정 — 판단직 프로토콜 §1)

1. **Ryn 73%의 지배 원인은 벽시계 사살(마찰)이다** — 운영 실측 진술 기반. P0-5로 도입된 insights 실패 유형 분해(`ab75bc6`)로 착수 전 1커맨드 재검증 가능.

## 1. 혁신 후보 4건

### R-1. 런 연속체 (Run Continuum) — "사살의 폐지"

**구현됨** (2026-07-17, Ryn): `forge run --continue {run_id}` — lib/agent-runner.sh 체크포인트 기록(`_agent_checkpoint_write`/`_agent_checkpoint_read`) + growth-log `result=checkpoint/exhausted, slice={n}` 필드. bats 회귀는 후속(Zen) 예정.

- **(a) 정의**: timeout/turn_cap 사살을 "실패"가 아닌 **체크포인트 정산**으로 바꾸고, 다음 슬라이스가 동일 세션(`--resume`) + 동일 worktree로 작업을 승계한다. 런의 원자성을 폐지하고 "작업 = 지속체, 런 = 슬라이스"로 재정의.
- **(b) 왜 혁신인가**: 지금 엔진의 암묵 전제는 "런은 원자적 성공/실패"다. 그래서 1500~1800s 사살이 관례화됐고, 사살 = 산출물 소멸 = 호스트 수동 마감이다. 이 전제가 깨지면 **타임아웃이 실패 사유에서 중립 이벤트로 강등**되고, 대형 태스크를 "죽지 않을 크기로 미리 쪼개는" 부담 자체가 사라진다 — 쓰는 방식이 바뀐다. 정량 가설: **Ryn 73% → 88%+**(팀 평균 회복, 사살분이 실패에서 제외되므로), **호스트 수동 마감 0건/4주**(현재 대형 런마다 관례).
- **(c) 실현 경로 (재사용 극대화)**: ① agent-runner 사살 경로(rc 124 / turn_cap)에서 kill 직전 `session_id + worktree 경로 + git diff --stat + 미완 요지`를 `.golem/checkpoints/{run_id}.json`으로 기록 — **P0-4의 partial 정산 로직이 이미 git 실측을 뜬다, 저장만 추가**. ② `forge run --continue {run_id}`: `--resume {session_id}` 재소환 — **`--resume` 인프라·즉사 폴백(P2-2)·recency 포인터 전부 기존**(agent-runner.sh:564-598, 756-764). ③ 격리는 기존 worktree 모듈, 반복 감시는 mission-loop 스턱 디텍터 재사용. **신규는 체크포인트 파일 포맷 + `--continue` verb 하나.**
- **(d) 리스크/비용**: 식은 세션 resume가 루프에 빠져 슬라이스만 태울 위험(→ 슬라이스 상한 3 + 스턱 디텍터), 무한 이어달리기가 진짜 능력 문제를 은폐할 위험(→ 슬라이스 수를 growth-log에 기록해 insights가 감지).
- **(e) 규모**: **M**

### R-2. 실측 정책 엔진 — "장부가 스케줄러가 된다"

- **(a) 정의**: model-routing/turn-cap/폴백 SOUL 선택의 정적 테이블을 growth-log 실측 분포(SOUL×태스크유형별 성공률·소요·비용)로 대체하는 결정론 정책 레이어.
- **(b) 왜 혁신인가**: 현재 성장 엔진(랭크·뱃지·케미)은 **관상용에 가깝다** — 라우팅은 정적 if문이 한다. 실측이 정책을 결정하는 순간 "경험이 권한이 된다"는 세계관이 코드로 실현되고, 미해결인 **폴백 SOUL 정책**("specialty 겹침 0 → junior 첫번째"라는 임의 규칙)이 "이 유형을 실제로 잘한 SOUL"로 바뀐다. 사용자의 "장부·게이트로 무게중심 이동" 문제의식의 본체. 정량 가설: turn_cap 오사살 0(캡=p95 실측+여유분), 폴백 선택이 임의에서 근거 기반으로.
- **(c) 실현 경로**: growth-log 175건+ 파싱은 insights.sh가 이미 함 — 집계를 리포트가 아닌 **머신 판독 정책 파일**(`.golem/policy.json`)로 굽고, model-routing.sh/triage.sh가 존재 시 정적 테이블보다 우선 참조. LLM 호출 없음 — 이 프로젝트의 성공 패턴(결정론 라우팅) 반복.
- **(d) 리스크/비용**: **데이터 희소성** — 175건을 (SOUL×유형) 셀로 나누면 셀당 표본이 얇아 노이즈 과적합 위험. 더 치명적으로, **현재 장부는 사살을 실패로 기록하고 있어 오염돼 있다** — R-1이 선행되지 않으면 garbage in.
- **(e) 규모**: **M~L**

### R-3. 이기종 러너 페더레이션 — "게이트 10배"

- **(a) 정의**: 테스트/검증 게이트를 원격 러너(Mac)에 오프로드 — 실행은 로컬, 게이트만 원격 디스패치(git push + SSH 트리거 + 결과 회수).
- **(b) 왜 혁신인가**: 게이트 30~90분 → 3~8분(~10x)이면 "검증은 하루 한 번 몰아서"가 "매 슬라이스마다"로 바뀐다 — 검증 빈도의 상전이. verify가 사살 대상에서 상시 게이트로.
- **(c) 실현 경로**: verify.sh의 러너 감지에 remote 백엔드 추가, worktree가 이미 격리 단위를 제공. 프로토콜은 "커밋 해시 + 테스트 명령 → exit code + 로그" 최소 계약.
- **(d) 리스크/비용**: **하드웨어 미확보(외부 의존) — 지금 착수 불가능이 최대 리스크.** SSH/네트워크 운영면 추가, macOS/Windows 결과 불일치(portability 감사로 상당 완화됨), 레포 내 자가검증 불가.
- **(e) 규모**: **L**

### R-4. 계약-정산 반전 (Ledger-first Inversion) — "지시를 걷어내고 장부만 남긴다"

- **(a) 정의**: 실행 전 **스코프 계약**(대상 파일 목록 + DoD)을 선언시키고, 실행 후 git 실측으로 정산하며, 일탈(선언 외 파일 수정·미선언 일탈)을 게이트가 결정론 판정 — 대신 오케스트레이션 프롬프트(절차 지시)는 랭크 상승과 무관하게 전반적으로 박화.
- **(b) 왜 혁신인가**: 엔진의 정체성이 "어떻게 하라고 지시하는 오케스트레이터"에서 "**무엇을 했는지 정산하는 거래소**"로 반전된다. 신형 모델이 하네스를 잘 다룰수록 지시 레이어는 감가상각되지만 장부·게이트는 모델이 좋아질수록 가치가 오른다. 실측 관측된 "대상 파일 일탈 시 무선언" 문제가 게이트로 흡수된다. 정량 가설: 일탈 감지율 0% → 100%, 프롬프트 정적 블록 토큰 감소.
- **(c) 실현 경로**: rubric 사전 계약(B-5)이 이미 "DoD 합의→주입→동일 항목 채점"을 뚫어놨다 — 같은 메커니즘에 `scope: [files]` 필드를 추가하고, P0-4 git diff 정산이 선언 대비 대조만 하면 된다.
- **(d) 리스크/비용**: 지시 박화의 효과 검증이 어렵다(eval 골든 태스크로 A/B 필요). 스코프 선언이 탐색적 태스크에 족쇄가 될 수 있음("SOUL은 족쇄가 아니라 나침반" 철학과 긴장) — 일탈을 차단이 아닌 **선언 의무**로 설계해야 함.
- **(e) 규모**: **L** (단, 스코프 게이트 단독이면 M)

## 2. 비교와 1순위 추천 (대안 기각 사유 — 판단직 프로토콜 §2)

| | 마찰 직결도 | 부품 재사용 | 지금 착수 가능 | 다른 후보에의 파급 |
|---|---|---|---|---|
| **R-1 런 연속체** | ◎ (사살 = 최대 실측 마찰) | ◎ (신규 최소) | ◎ | R-2의 데이터 정화, R-3의 긴급도 완화 |

**1순위: R-1 런 연속체.**

이유: (1) 실측 마찰의 최대 항목(벽시계 사살 관례화, Ryn 73%, 수동 마감)을 정면 타격한다. (2) 재사용률이 가장 높다 — `--resume` 인프라, 즉사 폴백, GOLEM_DONE partial 정산, worktree, 스턱 디텍터가 전부 가동 중이고 신규는 체크포인트 파일 + verb 하나다. (3) **다른 후보들의 전제 조건이다**: R-2는 사살이 실패로 기록되는 한 장부가 오염돼 있고, R-3은 게이트가 빨라져도 사살 시 산출물 소멸 문제가 남는다. R-1은 사살의 의미론 자체를 고쳐 둘 다의 기반을 놓는다. 그리고 이것은 R-4(장부 중심 반전)의 첫 구체 실현이기도 하다 — "정산 가능한 부분 산출물"이야말로 장부가 런 원자성을 깨는 지점이다.

**대안 기각 사유**:
- **R-3 기각**: 벽시계 이득의 절대량은 최대(10x)지만 Mac 하드웨어라는 외부 의존에 막혀 지금 착수 불가하고, 레포 안에서 자가검증할 수 없다. 또한 게이트가 빨라져도 "사살 = 소멸" 의미론은 그대로다. **하드웨어 확보 시 R-1 다음 최우선으로 승격 권고.**
- **R-2 기각(연기)**: 방향은 옳으나 현재 장부는 사살을 실패로 기록하는 오염 상태 + 175건을 셀로 나누면 표본이 얇다. R-1이 의미론을 정화하고 4~8주 실측이 더 쌓인 뒤가 적기다.
- **R-4 기각(부분 채택)**: 사용자 문제의식의 본체지만 전면 반전은 효과 검증이 어려운 L급 재설계다. 그 가치는 R-1(부분 정산)·R-2(실측 정책) 같은 구체 인스턴스를 통해 도착한다 — 우산 프레임으로 유지하되 첫 삽은 아니다. 단, **스코프 선언 게이트(M급 조각)는 R-1 직후 트랙으로 분리 추진 가치가 있다** ("대상 파일 일탈 무선언" 실측 문제 직결).

**이 판단의 리스크 상위 3 (판단직 프로토콜 §3)**:
1. **원인 오진**: Ryn 73%의 지배 원인이 사살이 아니라면(예: 능력/컨텍스트 문제) R-1의 효과가 반감된다 — 착수 전 `forge insights ryn`(P0-5 실패 유형 분해)으로 timeout/turn_cap 비중 확인이 선행 조건.
2. **식은 세션의 품질**: kill된 세션의 `--resume` 이어달리기가 신선한 재시작보다 오히려 못할 수 있다(누적 컨텍스트의 관성) — 슬라이스 상한 3 + eval 골든 태스크로 "이어달리기 vs 재시작" 1회 비교 측정을 성공 기준에 포함.
3. **가드 무력화의 재연**: 체크포인트가 "무한 관용"으로 오용되면 turn cap 무력화의 전철(C-2 이전 GOLEM_TURN_CAP_ENFORCE=0 관례)을 밟는다 — 슬라이스 상한 초과 시 `result=exhausted`로 명시 실패시켜 장부에 남긴다.

## 3. 1순위 착수 첫 스텝 (forge build 태스크 문장)

선행 1커맨드 (착수 게이트): `forge insights ryn` — 실패 중 timeout/turn_cap 비중 ≥ 50% 확인. 미달 시 이 플랜 재검토.

```
forge build: agent-runner의 벽시계/턴캡 사살 경로에 체크포인트 정산과 이어달리기를 추가하라 —
(1) kill 직전 session_id·작업 디렉토리(worktree 경로)·git diff --stat·GOLEM_DONE 부분 마커 유무를
    .golem/checkpoints/{run_id}.json 에 기록 (P0-4 partial 정산 로직 재사용),
(2) forge run --continue {run_id} 신설: 체크포인트의 session_id 로 --resume 재소환 +
    체크포인트 요지(완료분/미완분)를 태스크 블록에 주입, 슬라이스 상한 3 초과 시 result=exhausted,
(3) growth-log 에 result=checkpoint 와 slice={n} 필드 추가, 최종 슬라이스에서 git 실측 통합 정산,
(4) fake claude bats 회귀: 사살→체크포인트 기록→--continue 이어달리기→통합 정산→상한 초과 exhausted
    4경로. 담당: Ryn(agent-runner/checkpoint) + Zen(bats 회귀), Zen 크로스 리뷰 필수.
```

## 자가 반박
